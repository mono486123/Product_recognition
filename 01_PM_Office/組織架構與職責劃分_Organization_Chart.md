本專案採用「一人分飾多角」的公司模擬模式，將整體開發流程拆分為四個核心部門，以降低混亂並確保每一階段都有明確產出。

遇到的問題 / 任務,負責部門,解決核心邏輯
「這罐飲料認不出來，信心度只有 0.3」,AI 研發部,檢查 Dataset 是否缺少該角度、光線，或是標註有誤。
「辨識出商品了，但不知道多少錢」,專案管理部 (PM),檢查 SKU_List（Excel/JSON）是否有對應的 ID 與價格。
「阿嬤說手機上的字太小，看不清楚」,產品設計部,修改 Figma 稿，放大字體與對比度，重新定義 UI 色彩。
「按相機快門後，畫面卡住三秒才動」,軟體工程部,優化異步處理（Async），檢查是否在 UI Thread 執行了重度運算。
「模型檔案 50MB 太大，App 跑不動」,AI 研發部,進行模型量化（Quantization），將模型壓縮至 TFLite。
「想增加『買二送一』的促銷功能」,專案管理部 (PM),定義商業邏輯規格，決定這功能是否為 MVP (最小可行產品)。

一、 系統設計核心原理

本系統基於「AI 輔助、人工最後把關」的邏輯，不追求 100% 的機器完美，而是追求「總體效率」的提升。



1. 技術框架

核心算法： 使用 YOLOv11 (You Only Look Once) 物件偵測架構，具備高速度與高精準度。

訓練策略： 全量訓練 (Full Training)，針對 101 個 SKU 進行標記與學習，解決相似品差異問題。

部署環境： 採手機端或輕量化本地主機處理，減少設備發燙，非使用時可即時關閉。

2. 硬體與環境假設

拍攝場景： 固定於收銀台。

光線條件： 穩定光源，無需額外考慮極端暗光環境。

擺放邏輯： 允許商品在收銀台內隨機擺放（多位置數據增強）。

二、 業務邏輯流程 (Workflow)

系統採取「拍、辨、改、收」四步驟：



拍 (Capture)： 結帳時，將商品放置於收銀台，手機拍下一張全景照。

辨 (Inference)： YOLOv11 識別所有商品及其 SKU ID。

改 (Correction)： * 人工減法： 點擊誤判商品即可從清單「剪掉」。

人工加法： 沒辨識到的商品，透過內建計算機手動加入。

收 (Totaling)： 系統自動加總最終金額，完成結帳。

三、 功能模組詳解

1. AI 辨識模組 (Python Core)

負責載入權重並產出 JSON 格式的商品清單。



相似品解決方案： 針對包裝雷同的產品，在數據採集階段增加「細節特寫」的 Image Data。

2. UI/UX 簡單化介面

設計原則為「大按鈕、少層級」。



視覺反饋： 辨識出的商品會在螢幕照片上用「框」圈起來，方便一眼看出遺漏。

清單視窗： 列出 商品名 | 單價 | 刪除按鈕。

3. 內建計算機模組 (Manual Adjuster)

這是系統的「容錯防護網」。



提供數字鍵盤，供操作人員直接輸入「金額」而非搜尋品項（加快速度）。

具備 Total = AI_Total - Manual_Delete + Manual_Add 的運算邏輯。

四、 實施步驟與開發階段

階段 1：數據採集 (Data Preparation)

101 類 SKU： 每一類在收銀台進行 360 度旋轉擺拍。

標記： 使用 LabelImg 或 Roboflow 標出 Box。

階段 2：本地訓練 (Local Training)

使用 YOLOv11 預訓練權重。

在本地配置 GPU 環境進行 Train，輸出 best.pt。

階段 3：系統整合與 UI 開發

建立一個簡易的 Python (或是 Mobile App) 殼。
