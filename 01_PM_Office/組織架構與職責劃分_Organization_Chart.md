

# 🏢 專案組織架構與職責劃分 (Organization Chart)

本專案採用「一人分飾多角」的公司模擬模式，將開發流程拆分為四個核心部門，確保每一階段都有明確的目標與產出，並透過「部門決策」解決卡關問題。

---

## 🛠️ 跨部門問題解決清單 (Issue Handling)

| 遇到的問題 / 任務 | 負責部門 | 解決核心邏輯 |
| --- | --- | --- |
| 「這罐飲料認不出來，信心度只有 0.3」 | **AI 研發部** | 檢查 Dataset 是否缺少該角度、光線，或標註有誤。 |
| 「辨識出商品了，但不知道多少錢」 | **專案管理部 (PM)** | 檢查 SKU_List (Excel/JSON) 是否有對應 ID 與價格。 |
| 「使用者說手機字太小，看不清楚」 | **產品設計部** | 修改 Figma 稿，放大字體與對比度，優化 UI 色彩。 |
| 「按下快門後，畫面卡住三秒才動」 | **軟體工程部** | 優化異步處理 (Async)，避免在 UI Thread 執行運算。 |
| 「模型檔案 50MB 太大，App 跑不動」 | **AI 研發部** | 進行模型量化 (Quantization)，壓縮至 TFLite。 |
| 「想增加『買二送一』的促銷功能」 | **專案管理部 (PM)** | 定義商業邏輯規格，判斷是否為 MVP 核心功能。 |

---

## 1️⃣ 專案管理部 (PMO) — 大腦

**角色**：Project Manager (PM)

**核心關注**：時程、規格、驗收、使用者體驗（UX）。

### 💡 核心心態

* 冷靜、現實、結果導向。
* 「功能是否必要」優於「技術是否酷炫」。

### 📋 具體工作

* **規格凍結**：撰寫與維護 PRD，防止需求無限膨脹。
* **資料管理**：維護 `Master_SKU_List.xlsx`。
* **進度紀錄**：每日開會日誌與決策紀錄。

### 📅 PM 辦公室決策日誌 (開會紀錄)

* **12/17**：AI 轉型決策。因 TFLite 導出受阻，決議先轉 **.ONNX**，並產出 TOP_10 初版權重。
* **12/18**：軟體部交付 TOP_10 App 初版。PM 決定與設計部合作優化 UI 後再進行全樣品訓練。
* **12/21**：完成 V2 全商品辨識系統介面設計。
* **12/25**：結束所有商品拍攝任務，下令 AI 研發部進入 Train 階段。
* **12/26**：針對數據不平衡問題，決議進行資料清洗而非盲目訓練。
* **12/29：【重大轉向決策】**
* 判定「海量低品質圖片」無助於辨識率，暫以「選單法」保底。
* **技術升級**：要求研發部棄用 Nano 模型，改採 **YOLOv11s/m**。
* **範圍聚焦**：改為專攻「冰箱 31 類飲料」，設定驗收標準為 **Precision: 93% / Recall: 90%**。
* **轉型時程**：要求 12/31 結束 AI 階段，轉向 **SKU 管理系統** 開發。

## 2️⃣ 產品設計部 (Design) — 體驗

**角色**：UI / UX Designer

**核心關注**：大按鈕、高對比、減少層級（阿嬤也要會用）。

* **具體工作**：Figma 原型製作、定義 UI Flow。
* **紀錄產出**：`Design_Mockup_v1.png`、顏色規範表。

---

## 3️⃣ AI 研發部 (R&D - AI) — 技術

**角色**：AI 工程師 / 標註員

**核心關注**：模型精準度、推理速度、數據品質。

* **具體工作**：數據採集、Roboflow 標註、YOLO 訓練、格式轉換。
* **紀錄產出**：`data.yaml`、訓練報告、`best.onnx` / `best.tflite`。

---

## 4️⃣ 軟體工程部 (Engineering) — 實作

**角色**：Flutter App Developer

**核心關注**：效能穩定、Firebase 整合、商業邏輯實作。

* **具體工作**：Flutter 專案開發、Camera 串接、計算機與購物車邏輯。
* **紀錄產出**：Git Commit Log、技術文檔。
